C# Coding Conventions

The general rule we follow is “use .Net Core coding standards”. Below is general guidelines, if you have specific questions talk with the Architecture team or your team lead.

Naming Conventions

  • Avoid abbreviations unless it is a common abbreviation. (e.g. Id, Cpu, Atlas)
  • Use _camelCase for internal, protected, and private fields.
    • Prefix instance fields with _
      • Including private readonly or private static instances
    • When used on static fields, readonly should come after static (i.e. static readonly not readonly static).
  • All constant variables/fields should use PascalCasing. (regardless of the access modifier)
  • All public variables should use PascalCasing.
  • All methods should use PascalCasing.
    • One exception: local methods (from C# 7) should use the same rules as other method-scoped variables and be camelCased.

Variable Declarations

  • Do not use default(T) syntax in assignments.
  • Always use readonly when possible.
    • This includes making properties readonly (see example).
    • See the Readonly Property Examples.
  • Always specify the visibility modifier, even if it’s the default.
    • Always prefer private unless there’s a need to expose the method/variable/etc.
    • Visibility should be the first modifier (i.e. public abstract not abstract public).
  • Use language keywords instead of BCL types for both type references as well as method calls.
    • Use int, string, float instead of Int32, String, Single, etc) for both type references as well as method calls
    • Use int.Parse instead of Int32.Parse

General Guidelines

  • Use Allman style braces, where each brace begins on a new line.
    • A single line statement block can go without braces but the block must be properly indented on its own line and it must not be nested in other statement blocks that use braces.
    • For an if / else block, the entire block should be consistent on the use of braces. If the else block has braces, then the if block should as well, and vice-versa.
  • Use four spaces of indentation (no tabs).
  • Use nameof(...) instead of "..." whenever possible and relevant.
  • When including non-ASCII characters in the source code use Unicode escape sequences (\uXXXX) instead of literal characters. Literal non-ASCII characters occasionally get garbled by a tool or editor.

Constructors Guidelines

  • Only accept what is absolutely needed for the class to function properly.
    • Avoid dependency wrappers.
  • All parameters in a constructor should be assumed to be required unless one of the exceptions is met:
    • An applicable override exist where the parameter is not available.
    • The constructor specifies a default value for the constructor parameter.
    • Do not use default parameters on any public constructors.
  • The constructor should always throw a new ArgumentNullException when a parameter is required.
    • See Constructor Parameter Examples.

File Structure

  • Only a single class/enum per file.
  • At the class level, there should be an organization structure behind the code:
    • Private variables
    • Constructors
    • Public Getters
    • Public/Private Methods
    • Private / Public methods can be grouped by purpose or by modifier.

Obsolete Guidelines

  • Do not mark any object obsolete until there is an alternative in place.
  • In the comment for the obsolete attribute, include the Ticket Number that made this item obsolete along with a brief description of what to use instead.
  • When marking an object obsolete, always create another task to go back and remove after a reasonable period.
  • Never use obsolete as an alternative for a TODO.

TODO Guidelines

  • As a general practice, you should not leave TODO‘s in the code that will be committed to master.
  • Any TODO comment will almost always be lost in the noise and is impossible for Project Managers to see. This will lead to the TODO‘s never being completed, and is why they should be avoided.
  • If committing a TODO to master is required, then it should always have a ticket number associated, and should be present in the comment.
    • If you find a TODO comment without a ticket number, create it.
    • i.e. // TODO: TFS1234 - Need to refactor this section to handle errors better

Commit Guidelines

  • Ensure all code has been tested manually and all unit tests succeed before submitting a pull request.
  • Ensure the PR Description is clear/concise.
  • Ensure no new warnings were created.
  • Try to link to the work item in at least 1 commit like so: #12345
